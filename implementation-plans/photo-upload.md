# 写真アップロード機能 実装計画

このフォルダ（`implementation-plans/`）は、アプリの改修計画をメモしておくための場所。`docs/` とは別で、`app.js` の `docs` 一覧には入れず、学び（学習）タブには表示しない。

## 目的

フットサルの当日記録に、写真を1枚添付できるようにする。

- 記録一覧ではサムネイル表示
- サムネイルをタップすると写真の詳細（拡大表示）が見られる
- 詳細画面には「＜ 戻る」で記録一覧（または記録詳細）に戻れる導線をつける

## スコープ

- 1記録につき写真は **1枚まで**
- 対象は「当日記録」（`saveCurrentRecord` で保存する確定記録）。下書き（`futsal_drafts`）への添付は今回のスコープ外でよいか要確認
- 既存の記録（写真なし）は引き続き正常に表示できること

## 保存先（未決定・要相談）

会話の中でまだ決めていない。次回、以下のどちらかを選ぶところから再開する。

### 案A: Supabase Storage（推奨）
- 複数端末で同じ写真が見られる
- 事前にSupabaseダッシュボードでStorageバケットを1つ作成し、そのバケットに対して
  「本人の記録に紐づくファイルだけ読み書きできる」RLS相当のポリシーを設定する必要がある
- `futsal_records.data` にはファイルの公開URL（またはバケット内パス）だけを保存する
- アップロード/削除時にSupabase Storageへのアップロード処理・エラーハンドリングが増える

### 案B: ブラウザ内のみ（ローカル、Supabase不要）
- 設定不要ですぐ使える
- 写真はその端末のローカル記録にのみ保存（`localStorage`、base64エンコードして`data`に埋め込む想定）
- 他端末には表示されない。写真のサイズによっては`localStorage`の容量上限（多くのブラウザで5〜10MB程度）に早めに達する
- 案Aへ後から移行する場合、既存データの移行方法を別途考える必要がある

どちらにするかで、実装量とテスト観点がかなり変わるため、次回最初に決める。

## 想定するデータ構造

`normalizeRecord()` が返すレコードのオブジェクトに1フィールド追加する案:

```js
{
  // ...既存フィールド
  photo: {
    // 案Aの場合
    url: "https://.../storage/v1/object/public/xxx/yyy.jpg",
    // 案Bの場合
    // dataUrl: "data:image/jpeg;base64,...."
  } | null,
}
```

（キー名・構造は実装時に決め直してよい。互換性のため、写真なしレコードは `photo: null` もしくはフィールド自体が存在しない状態を許容する。）

## UI変更点（想定）

1. **記録フォーム（`#formView`）**
   - 「フットサルノートを記録」セクション付近に `<input type="file" accept="image/*" capture="environment">` を追加
   - 選択直後にプレビュー用サムネイルを表示（案Bなら`FileReader`でdata URL化、案Aなら選択のみ保持しておき保存時にアップロード）

2. **記録一覧（`#records` / `renderRecords`）**
   - 各記録カードに、写真があればサムネイル画像を表示
   - サムネイルをタップ/クリックすると写真詳細ビューを開く

3. **写真詳細ビュー（新規）**
   - 新しい `.view` セクション（例: `#photoDetailView`）を追加し、`navigate("#/photo/<recordId>")` のようなルートで遷移
   - 大きい画像 + 「＜ 戻る」ボタン（呼び出し元の一覧 or 記録詳細に戻る）
   - 既存の `docsDetailView` の「学び一覧へ戻る」ボタンや、`detailView` の「一覧へ戻る」ボタンと同じパターンを踏襲する

4. **記録詳細（`#detailView` / `renderDetail`）**
   - 記録詳細画面自体にも写真を表示し、タップで詳細ビューを開けるようにする（一覧・詳細どちらから見ても写真を開けるように）

## 実装ステップ（案A: Supabase Storage を選んだ場合）

1. Supabaseダッシュボードでバケット作成（例: `record-photos`）、本人の記録にのみ読み書きできるポリシーを設定
2. `app.js` にアップロード関数を追加（`supabaseClient.storage.from("record-photos").upload(...)`）
3. フォームに写真選択UIを追加し、保存時にアップロード → 公開URL取得 → `record.photo.url` に保存
4. 記録一覧・詳細・写真詳細ビューのレンダリングとルーティングを追加
5. 写真削除（記録削除時にStorage側のファイルも削除する必要があるか検討)

## 実装ステップ（案B: ローカルのみを選んだ場合）

1. フォームに写真選択UIを追加し、`FileReader` でbase64の data URL に変換して保持
2. 保存時に `record.photo.dataUrl` として `localStorage` の記録データに含める
3. 記録一覧・詳細・写真詳細ビューのレンダリングとルーティングを追加
4. 大きすぎる画像はアップロード前にcanvasでリサイズ/圧縮する処理を検討（容量対策）
