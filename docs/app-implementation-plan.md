# Webアプリ実装計画

## 目的

このWebアプリは、週末フットサルを単に記録する場所ではなく、前の晩や当日からピークパフォーマンスに近づくための準備と、活動後のフットサルノートを1セットで残す場所にする。

記録は次の2フェーズで扱う。

1. 準備: 前の晩や当日に、生活、心理、集中、価値を整える
2. 振り返り: 週末の活動後に、フットサルノートとして状況判断能力を高める記録を残す

準備フェーズは途中保存できることが重要。前の晩に書き始め、当日の朝や移動前に追記し、活動後に振り返りを同じ記録へ足して完成させる。

## 現状の課題

現在のアプリはブラウザの `localStorage` に記録を保存している。これは静的アプリとしては簡単だが、スマホ2台やPCから同じ記録へアクセスする用途には合わない。

課題:

- 端末ごとに記録が分かれる
- スマホで書いた記録をPCで見られない
- PCで書いた記録をスマホで続きから編集できない
- ブラウザのデータ削除で記録が消える可能性がある
- 長期的なフットサルノートとしては保存先が弱い

そのため、将来的にはサーバー側DBへ保存し、認証された同じユーザーならどの端末からでも同じ記録へアクセスできる構成にする。

## 目指す画面構成

### 準備フェーズ

前の晩や当日、フットサル場へ向かう前に使う。

目的:

- ピークパフォーマンスに近づく
- 活動に求める価値を確認する
- 生活、心理、集中を整える
- 不安や焦りを扱いやすくする
- プレー内容を細かく決めすぎず、状態を整える

入力項目案:

- 準備時の身体スコア
- 準備時の心スコア
- 今の状態メモ
- 生活準備: 睡眠、食事、水分、移動、荷物
- 気持ちの準備: ポジティブでリラックスした状態、自分への言葉
- 集中の準備: 今日の価値、役割、到着までに整えること
- インサイドルール: 自分との小さな約束

クイック入力例:

- 水分を取る
- 軽く食べる
- 早めに出る
- 呼吸を整える
- 役割を思い出す
- ポジティブに過ごす

### 振り返りフェーズ

週末の活動後に使う。ここはフットサルノートとして、状況判断能力、ゲームインテリジェンスの向上に直結する記録にする。

目的:

- ただの感想で終わらせない
- 「なぜ」を深掘りする
- 認知の穴を見つける
- 次回の判断を具体化する
- 短い合言葉に圧縮し、次の準備へ戻す

入力項目案:

- 活動後の身体スコア
- 活動後の心スコア
- 活動後の状態メモ
- 場面: どんな状況だったか
- 結果: 何が起きたか
- なぜ: なぜそうなったか
- 認知の穴: 何を見ていなかったか、何に気づけなかったか
- 次の選択肢: もし同じ状況ならどうするか
- 合言葉: ピッチ上で一瞬で思い出す短いキュー
- できたこと
- 次回の準備へ戻すこと

## フットサルノートのPDCA

振り返りフェーズは、状況判断能力を上げるためにPDCAで扱う。

### Check

結果だけでなく「なぜ」を見る。

例:

- パスミスをした
- トラップしてから相手を探していた
- 受ける前に逆サイドを見ていなかった
- 顔が下がっていた

ここでは「技術不足」とまとめず、どの情報を見落としていたかを言語化する。

### Action / Plan

次回の選択肢を事前に作る。

例:

- 相手が寄せてきたらワンタッチで戻す
- 空いていたら前を向く
- 受ける前に逆サイドを見る
- パスをもらう前に体の向きを作る

「次は周りを見る」だけでなく、「もしAならB、CならD」の形にする。

### Cue

改善策を、試合中に思い出せる短い合言葉にする。

例:

- 受ける前に見る
- 見る、向く、運ぶ
- 危険、機会、管理
- 曲がる前に見る

長い反省はピッチ上では使いにくい。ノートでは、次回の実行に使える短い言葉まで圧縮する。

### Do

次回の活動で試す。できた、できなかったを繰り返し、意識的な判断を少しずつ自動化する。

## データ設計

最初は項目が変わりやすいため、DBでは記録内容をJSONとして持つのが現実的。あとから検索したい項目だけ列として切り出す。

想定テーブル:

```sql
users
- id
- google_sub
- email
- created_at
- updated_at

records
- id
- user_id
- date_time
- preparation_json
- reflection_json
- created_at
- updated_at
```

`records.user_id` によって、自分の記録だけを取得する。

## 認証方針

一人用でも認証は入れる。スマホ2台、PCなど複数端末から同じ記録へアクセスするため。

自前IDとパスワードは避け、Googleアカウント認証を優先する。

理由:

- パスワードを自作管理しなくてよい
- スマホ2台で同じGoogleアカウントを使える
- パスワードリセットやハッシュ化を自前実装しなくてよい
- 将来ユーザーが増えても拡張しやすい

## インフラ候補

### 採用: Supabase

今回の用途には、Supabaseが最も早い候補なので、まずSupabaseで同期を実装する。

- PostgreSQL
- Google認証
- Row Level Security
- JavaScript SDK
- フロントから扱いやすい

個人用の記録アプリとして、DBと認証をまとめて始めやすい。

### 代替候補: Railway + PostgreSQL + API

Railwayを使う場合は、ブラウザからSQLへ直接つながず、必ずAPIを挟む。

構成:

```text
スマホ / PC
  -> Webアプリ
  -> APIサーバー
  -> PostgreSQL
```

API案:

- `GET /api/records`
- `POST /api/records`
- `PUT /api/records/:id`
- `DELETE /api/records/:id`

自由度は高いが、認証、セッション、API、DB接続を自分で実装する範囲が増える。

## 実装ステップ

1. 現在のローカル保存版を2フェーズ構成に直す
2. データ構造を `preparation` と `reflection` に分ける
3. 既存データを壊さないように移行処理を入れる
4. 詳細表示、一覧、Markdown/JSONエクスポートを新構造に対応する
5. 保存処理を `storage` モジュールとして分離する
6. まずは `localStorage` 実装を残す
7. SupabaseまたはRailway用の保存実装へ差し替える
8. Google認証を入れる
9. ログイン中ユーザーの記録だけを取得、保存、削除できるようにする
10. スマホ2台とPCで同じ記録が見えることを確認する

## 最初に実装する範囲

まずはDB化の前に、アプリの意味と保存データを正しくする。その後、SupabaseのGoogle認証とPostgreSQL保存に対応する。

最初の実装範囲:

- 画面を準備フェーズと振り返りフェーズに分ける
- 準備フェーズの文言をピークパフォーマンス準備に寄せる
- 振り返りフェーズにフットサルノート項目を追加する
- 途中保存を維持する
- 一覧、詳細、エクスポートを新項目に対応する

その後、保存先をサーバーDBへ移行する。
